# StockBrief MVP PRD

## 1. Product Summary

StockBrief is a 근거 기반 국내 주식 종목 후보 추천 서비스. It helps users review Korean listed companies by showing deterministic recommendation candidates, score components, evidence, data freshness, missing data, and risk tags.

Recommendation means `검토 후보 추천`. It is not a trading instruction, investment advisory service, or promise of future returns.

## 2. Product Principles

- Evidence-first: every candidate must expose source-backed evidence.
- Deterministic scoring: scores are computed by the backend score engine, not generated by AI.
- Neutral language: product copy must use review-oriented wording.
- Guest-first MVP: users can use the MVP without login or signup.
- Transparent uncertainty: missing or stale data must be shown directly.

## 3. Financial Wording Policy

Allowed wording:

- `추천 후보`
- `추천 이유`
- `오늘의 추천 후보`
- `검토해볼 수 있습니다`
- `확인이 필요합니다`
- `공개 데이터 기준입니다`

Prohibited user-facing wording:

- `매수`
- `매도`
- `목표가`
- `진입가`
- `손절가`
- `수익 보장`
- `확실`
- `무조건`

The product may explain why a stock appears as a review candidate, but it must not instruct the user to trade or imply certainty.

## 4. MVP Scope

### P0 Features

MVP 기능 고정 목록:

- 온보딩
- 추천 후보 리스트
- 추천 상세
- evidence 조회
- AI 추천 이유 설명
- localStorage 관심종목

| Feature | Description | Acceptance Criteria |
| --- | --- | --- |
| Onboarding | Explain that StockBrief provides evidence-based review candidates, not trading advice. | User sees neutral policy copy before or during first use. |
| Recommendation candidate list | Show today's domestic stock review candidates. | Each row includes `ticker`, company name, market, total score, evidence count, data freshness, and risk tags. |
| Recommendation detail | Show score breakdown, reasons, evidence, missing data, and risks for one ticker. | Detail view maps directly to `GET /v1/recommendations/candidates/{ticker}`. |
| Evidence view | Show source documents and evidence chunks used by the score/reason. | Each evidence item includes source name, URL when available, published date, fetched time, and evidence type. |
| AI reason explanation | Explain precomputed recommendation reasons and evidence in neutral language. | AI cites evidence IDs or source URLs and never creates its own score. |
| localStorage watchlist | Let guest users save 관심종목 locally. | Watchlist persists in browser `localStorage`; no server account is required. |
| Stock search | Search by ticker or company name. | Search uses `GET /v1/stocks/search`. |

### P1 Features

- Email-based login and signup through AWS Cognito.
- Email verification before normal authenticated use.
- Server-side user profile with Cognito `sub`, email, email verification status, and optional nickname.
- Server-side preferences.
- Multi-device watchlist sync through `/v1/me/watchlist`.
- Authenticated chat session history.
- Notification settings.
- KRX production ingestion if not already needed as fallback or seed data.

P1 must not remove or block the P0 guest experience. Users who are not logged in continue using localStorage watchlist and public recommendation APIs.

Server-side preferences use a simple whole-document save policy in P1. The account screen preserves fields from the latest loaded preferences snapshot when it saves known controls, but concurrent edits from another browser or device are resolved by last successful save. Field-level conflict handling is deferred until concurrent preference editing becomes a demonstrated product need.

### Out Of Scope For MVP

- Trading execution.
- Portfolio allocation.
- User-specific financial advice.
- Target price workflows.
- Entry or stop-loss workflows.
- Guaranteed return claims.
- LLM-generated scoring.

## 5. User Flows

### Flow A: First Visit And Onboarding

1. User opens the web app.
2. App shows concise onboarding: StockBrief provides public-data-based review candidates.
3. User continues to candidate list without authentication.
4. App stores onboarding completion in `localStorage`.

Example localStorage values:

```json
{
  "stockbrief:onboarding_completed": true,
  "stockbrief:watchlist": ["005930", "035420"],
  "stockbrief:preferences": {
    "markets": ["KOSPI", "KOSDAQ"],
    "risk_tags": ["data_gap", "high_volatility"]
  }
}
```

### Flow B: Review Candidate List

1. Frontend calls `GET /api/v1/recommendations/candidates`.
2. Next.js proxy forwards to backend `GET /v1/recommendations/candidates`.
3. If the user is authenticated and the URL has no explicit `risk_profile`, frontend initializes the candidate filter from `GET /v1/me/preferences`.
4. User reviews score, evidence count, freshness, and risk tags.
5. User opens a candidate detail page.

### Flow C: Recommendation Detail

1. Frontend calls `GET /api/v1/recommendations/candidates/{ticker}`.
2. Detail page displays:
   - total score
   - 8 component scores
   - recommendation reasons
   - evidence gate status
   - evidence list
   - risk tags
   - missing data
3. User may save the ticker to localStorage watchlist.

### Flow D: AI Explanation

1. User asks why a candidate appeared.
2. Frontend calls `POST /api/v1/chat`.
3. Backend answers using only the candidate score, reasons, and evidence available from the API.
4. If evidence is insufficient, AI says confirmation is required.

### Flow E: P1 Account And Watchlist Sync

1. User starts email login or signup from `/account`.
2. Cognito handles password input and email verification; StockBrief DB never stores password material.
3. After successful callback, frontend stores the short-lived auth session in browser session storage.
4. Frontend reads `stockbrief_watchlist_v1` from localStorage.
5. Frontend calls `GET /v1/me/watchlist`, removes duplicate tickers, then calls `POST /v1/me/watchlist/import` for server-missing items.
6. Frontend records sync state in `stockbrief_watchlist_v1_sync_state`.
7. Logged-in watchlist actions use server endpoints; logged-out actions continue using localStorage.

## 6. Evidence Gate

A stock is eligible for the MVP recommendation candidate list only when:

- It has at least 2 evidence items.
- It has at least 1 risk signal.
- It has a required data basis date in `data_freshness.as_of`.
- It returns `missing_data`, even when the array is empty.

If a ticker fails the evidence gate:

- It must not appear in the default candidate list.
- Its detail response may still be returned for debugging or search-driven review.
- The response must set `is_candidate_eligible` to `false` and explain which gate checks failed.

Evidence gate JSON example:

```json
{
  "is_candidate_eligible": true,
  "evidence_count": 3,
  "risk_count": 2,
  "data_freshness": {
    "as_of": "2026-06-09",
    "price_as_of": "2026-06-09",
    "financials_as_of": "2026-03-31",
    "news_fetched_at": "2026-06-09T08:30:00Z"
  },
  "missing_data": []
}
```

## 9. Success Criteria

- User can open the app and see recommendation candidates without authentication.
- Every candidate in the list passes the evidence gate.
- Detail pages expose score components, reasons, evidence, freshness, missing data, and risk tags.
- AI explanations cite evidence IDs or source URLs.
- Prohibited wording does not appear in user-facing copy except in policy documentation.
